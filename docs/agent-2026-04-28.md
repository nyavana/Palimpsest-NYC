Things confirmed end to end:

- Tool dispatch + retrieval ledger work — every citation references a `doc_id`
  that `search_places` actually returned.
- Citation verification passed (`verified: true`, `warning: null`).
- Server-side `plan_walk` resolved cited place_ids back to coordinates from
  postgres, built an ordered route, and emitted leg distances.
- SSE framing, event ordering, and final terminator all match the locked
  contract — `walk` precedes `done`, both follow `narration` and `citations`.
- The agent honestly acknowledges retrieval gaps in its narration ("the
  searches did not return the Cathedral of St. John the Divine") rather
  than inventing a citation. That's the V1 verifier doing its job.

## Performance note (kimi-k2.6 thinking-token cost)

End-to-end latency on the validation run was **354 seconds** with kimi-k2.6
across all three router tiers. Most of that is kimi's thinking-token
overhead — each turn spends 50-100s reasoning before emitting a tool call
or content. The §9 contract is correct, but the runtime cost is high for
an interactive demo.

The agent loop already does what it can without breaking the contract:

- Final turn strips the tool surface and forces `response_format=json` so
  the model can't get stuck in a tool-call refinement loop.
- Final-turn `max_tokens=8192` (vs 2048 for tool turns) — kimi was
  consuming the full 2048 on thinking and never reaching the JSON.
- Allows one extra retry turn beyond the 6-turn cap so the verification
  retry actually has a chance to run.

If demo latency becomes a blocker for §13, the simplest knob is to switch
`LOCAL_LLM_MODEL` to a non-thinking model (e.g., `google/gemma-4-31b-it`)
for the simple/standard tier; the router config supports per-tier model
overrides without code changes. That's already speced in
`swap-llm-tiers-and-lock-mvp-decisions/specs/llm-router/spec.md`.
